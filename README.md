# Machi Koro - OOP Project

Десктопное приложение для администрирования партии в настольную игру "Мачи Коро".

## 1. Ядро и Консольный интерфейс

### Реализованный функционал:
* **Интерактивное консольное лобби:** Добавление от 2-х игроков.
* **Игровой цикл (Game Loop):** Броски кубиков, покупка предприятий, постройка достопримечательностей, передача хода, досрочный выход из игры.
* **Строгая экономика и инварианты:** Игрок не может уйти в отрицательный баланс при штрафах, нельзя купить карты, на которые не хватает монет или которых нет на рынке.
* **Продвинутый Рынок:** Скрытая перемешанная колода и автоматическое пополнение резерва на столе до 5 карт после каждой покупки.
* **Автоматизация CI:** Настроен GitHub Actions для автоматического прогона тестов при пушах.
* **Тестирование:** Код покрыт Unit-тестами (домен), Интеграционными тестами (взаимодействие стратегий карт и движка) и Системными E2E-тестами (симуляция партии от начала до победы).

##  Архитектурные изменения относительно изначальной UML-диаграммы

В ходе реализации первой части проекта оригинальная архитектура была проанализирована и доработана:

1. **Добавлен паттерн Dependency Injection для броска кубика (Dice)**
   * *Проблема:* В оригинальной архитектуре движок сам генерировал случайные числа `(1..6).random()`. Это делало невозможным написание детерминированных интеграционных и системных тестов.
   * *Решение:* В слой Application добавлен интерфейс `Dice` с реализациями `RandomDice` (для игры) и `FakeDice` (для тестов). Кубик внедряется в `MachiKoroEngine` через конструктор.

2. **Введено понятие приоритета карт (Enum `CardColor`)**
   * *Проблема:* По правилам игры доходы и штрафы должны списываться в строгом порядке (Красные штрафы -> Зеленые/Синие доходы -> Фиолетовые эффекты). На UML не был предусмотрен механизм сортировки карт.
   * *Решение:* Создан Enum `CardColor(val priority: Int)`, по которому `MachiKoroEngine` сортирует собранные со стола карты перед вызовом `IncomeStrategy.calculateIncome`.

3. **Добавлена `PurpleIncomeStrategy`**
   * *Проблема:* В оригинальной UML (Domain Layer) присутствовали только зеленая, синяя и красная стратегии. Однако в игре есть фиолетовые карты "Крупного бизнеса" со сложными эффектами.
   * *Решение:* Добавлена недостающая стратегия для реализации спец-эффектов (например, "Стадион", забирающий монеты у всех остальных игроков), логика интегрирована в `TurnContext`.

4. **Изменена сигнатура метода `Player.deductCoins`**
   * *Проблема:* По правилам игрок не может уйти в отрицательный баланс. На UML метод ничего не возвращал, поэтому при красной карте (штраф) владелец кафе мог получить из "воздуха" больше монет, чем было у штрафника.
   * *Решение:* Метод `deductCoins(amount: Int)` теперь возвращает `Int` — количество *реально* списанных монет, которое затем начисляется владельцу карты.

5. **Инкапсуляция скрытой колоды в `Market`**
   * *Проблема:* Изначально предполагалось, что все доступные карты просто лежат в списке. В реальности игра требует закрытой колоды и ограниченного резерва на столе.
   * *Решение:* В класс `Market` добавлена внутренняя закрытая колода (`deck`). При создании игры она перемешивается, а метод `removeCard` теперь автоматически вызывает инкапсулированный метод `replenishMarket()`, поддерживая на столе всегда доступный резерв карт.
