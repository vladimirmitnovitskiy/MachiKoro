# Machi Koro - OOP Project

Десктопное приложение для администрирования партии в настольную игру "Мачи Коро".

## Архитектурные изменения относительно UML (Задание 1)

В ходе реализации первой части проекта оригинальная UML-диаграмма была доработана:

1. **Добавлен паттерн Dependency Injection для броска кубика (Dice)**
    * **Проблема:** В оригинальной архитектуре движок сам генерировал случайные числа `(1..6).random()`. Это делало невозможным написание детерминированных Unit и Интеграционных тестов.
    * **Решение:** В слой Application добавлен интерфейс `Dice` с реализациями `RandomDice` (для игры) и `FakeDice` (для тестов). Теперь кубик внедряется в `MachiKoroEngine` через конструктор.

2. **Введено понятие приоритета карт (CardColor)**
    * **Проблема:** По правилам игры доходы и штрафы должны списываться в строгом порядке (сначала красные, затем зеленые/синие, затем фиолетовые). На UML не был предусмотрен механизм сортировки карт.
    * **Решение:** Создан Enum `CardColor(val priority: Int)`, по которому `MachiKoroEngine` сортирует собранные карты перед вызовом `IncomeStrategy.calculateIncome`.

3. **Добавлена `PurpleIncomeStrategy`**
    * **Проблема:** В оригинальной UML (Domain Layer) присутствовали только зеленая, синяя и красная стратегии. Однако в игре есть фиолетовые карты "Крупного бизнеса" (например, Стадион или Телецентр), которые требуют отдельной логики.
    * **Решение:** Добавлена недостающая стратегия для реализации сложных спец-эффектов.

4. **Изменена сигнатура метода `Player.deductCoins`**
    * **Проблема:** По правилам игры, игрок не может уйти в отрицательный баланс при выплате штрафов (красные карты). На UML метод ничего не возвращал, поэтому владелец карты мог получить из воздуха больше монет, чем было у штрафника.
    * **Решение:** Метод `deductCoins(amount: Int)` теперь возвращает `Int` — количество реально списанных монет.
